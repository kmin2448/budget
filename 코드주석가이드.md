# COSS 예산관리 시스템 — 코드 주석 가이드

복잡한 비즈니스 로직과 예외 처리에서 **"왜 이렇게 짰는가"** 를 설명하는 문서입니다.
코드를 처음 보는 개발자가 같은 의문을 갖지 않도록 핵심 결정 사항을 기록합니다.

---

## 1. Google Drive 업로드 — 서비스 계정 대신 사용자 OAuth 토큰 사용

**파일:** `lib/google/drive.ts`, `app/api/drive/expenditure-upload/route.ts`

```typescript
// lib/google/drive.ts
function getDriveClient(accessToken: string) {
  const oauth2 = new google.auth.OAuth2(...);
  oauth2.setCredentials({ access_token: accessToken });
  return google.drive({ version: 'v3', auth: oauth2 });
}
```

**Why:**
Google Drive API는 서비스 계정으로 호출하면 서비스 계정 소유의 별도 Drive 공간에 파일이 생성됩니다.
이 공간은 무료 구글 계정과 동일하게 **15GB 용량 제한**이 있고, 구글 워크스페이스 없이는 용량 확장이 불가합니다.
반면, 로그인한 사용자의 OAuth 토큰으로 호출하면 **해당 사용자의 개인 Drive에 업로드**되므로,
조직 Drive 용량(보통 무제한) 정책을 그대로 따를 수 있습니다.

## 2. OAuth scope에 `drive.file`만 요청하는 이유

**파일:** `lib/auth.ts`

```typescript
scope: 'openid email profile https://www.googleapis.com/auth/drive.file',
```

**Why:**
`drive` scope(전체 접근)를 요청하면 GCP OAuth 심사를 통과해야 해서 배포가 지연됩니다.
`drive.file`은 **이 앱이 생성한 파일에만** 접근하는 제한적 scope라서 심사 없이 사용 가능합니다.
단, 앱 외부에서 업로드된 기존 파일에는 접근할 수 없으므로, 파일 삭제 시 Drive에서 못 찾아도 DB 레코드는 삭제하도록 예외 처리합니다.

## 3. `access_type: 'offline'` + `prompt: 'consent'` 강제 설정

**파일:** `lib/auth.ts`

```typescript
authorization: {
  params: {
    access_type: 'offline',
    prompt: 'consent',
  },
},
```

**Why:**
`access_type: 'offline'` 없이는 구글이 리프레시 토큰을 발급하지 않습니다.
리프레시 토큰이 없으면 액세스 토큰(1시간 유효) 만료 후 사용자가 다시 로그인해야 합니다.
`prompt: 'consent'`는 이미 OAuth를 허용한 사용자도 리프레시 토큰을 **재발급**받도록 강제합니다.
이를 생략하면 재로그인 시 리프레시 토큰이 발급되지 않아 Drive 업로드가 세션마다 끊깁니다.

## 4. 액세스 토큰 자동 갱신 로직

**파일:** `lib/auth.ts`

```typescript
async jwt({ token, user, account }) {
  // 토큰 유효하면 그대로 반환
  if (Date.now() < (token.accessTokenExpires as number ?? 0)) return token;
  // 만료 시 갱신
  return refreshAccessToken(token);
},
```

**Why:**
NextAuth JWT 콜백은 페이지 요청마다 실행됩니다.
만료 여부를 체크하지 않으면 매 요청마다 불필요하게 구글 토큰 갱신 API를 호출하게 됩니다.
`accessTokenExpires`를 밀리초 타임스탬프로 저장해 두고, 현재 시각과 비교해서 만료된 경우에만 갱신합니다.
갱신 실패 시에는 `error: 'RefreshAccessTokenError'`를 토큰에 심어서 클라이언트가 재로그인 안내를 띄울 수 있도록 합니다.

## 5. Google Sheets Named Range 상수화

**파일:** `constants/sheets.ts`

```typescript
export const NAMED_RANGES = {
  BUDGET_PLAN: '예산계획',
  ADJUSTMENT: '증감액',
  // ...
} as const;
```

**Why:**
Google Sheets Named Range 이름은 스프레드시트 내부에서 관리되는 문자열입니다.
코드 전반에서 한글 문자열을 직접 쓰면, 담당자가 스프레드시트의 Named Range명을 변경했을 때
어디를 고쳐야 할지 찾기 어렵습니다. 상수로 단일 관리하면 **한 곳만 수정**하면 됩니다.

## 6. 특수문자 시트명을 Named Range로만 참조하는 이유

**파일:** `app/api/budget/route.ts`

```typescript
// 특수문자(★)가 포함된 시트명을 직접 참조하면 API 오류 발생 → Named Range 사용
const currentRes = await sheets.spreadsheets.values.get({
  range: NAMED_RANGES.ADJUSTMENT,  // '증감액' Named Range
});
```

**Why:**
Google Sheets API에서 시트명에 `★` 같은 특수문자가 포함된 경우, `'★취합'!A1:B10` 형식의 직접 참조는
URL 인코딩 문제로 API 호출이 실패합니다. Named Range는 내부 ID로 참조되므로 이 문제를 회피할 수 있습니다.

## 7. 집행 내역 집계 키를 3단계(비목||세목||보조세목)로 구성하는 이유

**파일:** `app/api/budget/route.ts`

```typescript
// 3단계 키(비목||이나라세목||이나라보조세목)로 집계 — 2단계 키는 동일 비목+세목에 여러
// 보조세목이 존재할 때 중복 합산되는 버그를 유발함
const subKey = `${cat}||${subcat}||${subdetail}`;
```

**Why:**
같은 비목·세목 아래 보조세목이 여러 개인 경우, 2단계 키(`비목||세목`)로만 집계하면
해당 세목에 속한 모든 보조세목의 금액이 각 보조세목별 행에 중복으로 합산됩니다.
3단계 키로 집계해야 세목 하위 보조세목별로 집행액이 정확히 분리됩니다.

## 8. Sentinel 행으로 유효 데이터 범위 한정

**파일:** `app/api/budget/route.ts`

```typescript
const SENTINEL = '새로운 집행 내역 작성시 이 행 위로 작성';
const sentinelIdx = allExecRows.findIndex((row) =>
  row.some((cell) => String(cell ?? '').includes(SENTINEL)),
);
const validExecRows = sentinelIdx === -1 ? allExecRows : allExecRows.slice(0, sentinelIdx);
```

**Why:**
Sheets API로 범위(`A6:P500`)를 읽으면 Sentinel 아래의 빈 행이나 안내 문구까지 포함됩니다.
Sentinel 텍스트가 있는 행을 경계로 삼아 그 위쪽만 집계 대상으로 한정합니다.
이를 통해 실무자가 Sheets에 메모나 설명을 Sentinel 아래에 추가해도 집계가 오염되지 않습니다.

## 9. Drive 파일 삭제 실패를 무시하고 DB는 반드시 삭제

**파일:** `app/api/drive/expenditure-upload/route.ts`

```typescript
try {
  await deleteFromUserDrive({ accessToken: session.accessToken, fileId: existing.drive_file_id });
} catch {
  // Drive에 파일이 이미 없어도 DB 레코드는 삭제
}
await supabase.from('expenditure_files').delete()...;
```

**Why:**
사용자가 Google Drive에서 직접 파일을 삭제했거나, OAuth 토큰 scope가 달라진 경우
Drive API 삭제 호출이 실패(404 또는 403)할 수 있습니다.
이 경우 Drive 삭제 실패를 이유로 DB 레코드까지 남겨두면 "파일 없음" 상태가 영구 잔류합니다.
Drive 삭제는 best-effort로 시도하고, DB 정합성은 반드시 보장하는 방향으로 처리합니다.

## 10. 파일명 특수문자 치환

**파일:** `app/api/drive/expenditure-upload/route.ts`

```typescript
const safeName = file.name.replace(/[/\\:*?"<>|]/g, '_') || 'upload.pdf';
```

**Why:**
Windows/Mac 파일 시스템 예약 문자(`/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`)가 포함된 파일명은
Google Drive API 업로드 시 오류를 발생시키거나 파일명이 잘립니다.
업로드 전 서버에서 미리 치환해 클라이언트 OS에 무관하게 안전하게 처리합니다.

## 11. `super_admin` 권한 체크를 API마다 DB에서 재조회하는 이유

**파일:** `app/api/admin/users/route.ts`

```typescript
async function assertSuperAdmin(email: string) {
  const supabase = createServerSupabaseClient();
  const { data: user } = await supabase.from('users').select('role').eq('email', email).single();
  if (user?.role !== 'super_admin') throw new Error('권한 없음');
}
```

**Why:**
세션의 role은 JWT에 캐싱된 값으로, 로그인 이후 DB에서 역할이 변경되어도 세션 만료 전까지는 구 값이 남습니다.
관리자 API처럼 민감한 작업은 세션 캐시를 신뢰하지 않고 **매번 DB에서 최신 role을 확인**합니다.
일반 편집 권한 체크(`checkPermission`)도 동일한 이유로 DB를 직접 조회합니다.

## 12. 인건비만 월별 그룹화 없이 단순 목록으로 렌더링

**파일:** `components/expenditure/ExpenditureTable.tsx`

**Why:**
인건비는 월 단위 지급 구조가 아닌 인원·직급 기준으로 관리됩니다.
다른 비목은 3월~2월 12개 월 칼럼(`monthlyAmounts[12]`)으로 집행 시점을 구분하지만,
인건비에 이를 적용하면 담당자가 "몇 월에 줬는지"를 억지로 기입해야 하는 불필요한 입력이 발생합니다.
실무 요건 반영으로 인건비만 예외 처리합니다.

## 13. `ignoreDuplicates: true` — 재로그인 시 기존 사용자 정보 보호

**파일:** `lib/auth.ts`

```typescript
await supabase.from('users').upsert(
  { email: user.email, name: user.name ?? null },
  { onConflict: 'email', ignoreDuplicates: true },
);
```

**Why:**
`upsert`에서 `ignoreDuplicates: false`(기본값)를 쓰면 재로그인마다 `name`이 구글 프로필 값으로 덮어쓰입니다.
관리자가 Supabase에서 직접 이름을 수정해 놓은 경우 원복될 수 있습니다.
`ignoreDuplicates: true`로 설정해 **이메일이 이미 존재하면 아무 것도 변경하지 않도록** 합니다.
신규 사용자만 INSERT되고, 기존 사용자 정보는 보호됩니다.
